# PRD: 여사친/남사친 경계 테스트

## 1. 개요

### 1.1 프로덕트명
**Where is the Line? - 여사친/남사친 경계 테스트**

### 1.2 한 줄 요약
"이 행동, 친구니까 가능한 거야 아닌 거야?" — 이성 친구 사이의 경계를 테스트하고, 커플이 함께 비교할 수 있는 모바일 웹 서비스

### 1.3 목표
- 사용자가 이성 친구 관계에서의 경계를 재미있게 확인할 수 있다
- 커플 비교 기능으로 바이럴 공유를 유도한다
- 카카오톡 공유를 통해 자연스러운 트래픽 확산을 만든다

### 1.4 타겟 사용자
- 10~30대 남녀
- 연인 관계이거나 이성 친구가 있는 사용자
- 모바일 카카오톡 환경 사용자

---

## 2. 핵심 기능

### 2.1 개인 테스트

| 항목 | 설명 |
|------|------|
| 플랫폼 | 모바일 웹 (반응형) |
| 질문 수 | N개 (추후 확정) |
| 질문 형식 | 구체적인 상황 제시 + **가능 / 불가능** 2지선다 |
| 결과 | 답변 패턴 기반 개인 결과 타입 + 요약 문구 생성 |
| 공유 | 결과 화면에서 카카오톡 공유 링크 생성 |

**플로우:**
1. 사용자가 테스트 시작 버튼을 누른다
2. 질문이 순차적으로 표시된다 (상황 설명 + 가능/불가능 선택)
3. 모든 질문에 답변하면 결과 타입과 요약 문구가 화면에 표시된다
4. 카카오톡 공유 버튼으로 결과를 공유할 수 있다

### 2.2 커플 비교

| 항목 | 설명 |
|------|------|
| 진입 | 결과 화면의 "커플 비교" 버튼 |
| 매칭 방식 | 임시 room (세션 코드) 기반 |
| 데이터 저장 | 답변 배열만 저장 (개인정보 없음) |
| room 만료 | 생성 후 24~48시간 자동 삭제 |

**플로우:**
1. A가 "커플 비교" 버튼을 누른다
2. 시스템이 임시 room(세션 코드)을 생성한다
3. A의 답변 배열을 room에 저장한다
4. A가 room이 포함된 링크를 카카오톡으로 공유한다
5. B가 링크를 통해 진입하여 동일한 테스트를 진행한다
6. B의 답변을 같은 room에 저장한다
7. 두 사람의 답변이 모두 모이면 비교 결과를 생성한다

**비교 결과 구성:**
- 각자의 결과 타입
- 질문별 일치/불일치 현황
- "갈리는 포인트 TOP N" 요약
- 커플 비교 요약 문구

## 3. 서비스 아키텍처 원칙

### 3.1 데이터 전략
- **질문 데이터**: 정적 데이터로 클라이언트에 포함 (서버 호출 없음)
- **결과 타입/문구**: 정적 데이터로 클라이언트에 포함 (서버 호출 없음)
- **서버 역할**: 커플 비교용 room 생성 / 저장 / 조회만 담당

### 3.2 트래픽 대응
- 개인 테스트는 클라이언트 단독 처리 → 서버 부하 없음
- 트래픽 폭증 시 "커플 비교" 기능만 일시 제한
- 개인 결과 제공은 어떤 상황에서도 유지 (서비스 가용성 우선)

### 3.3 개인정보 보호
- room에는 답변 배열만 저장 (개인 식별 정보 없음)
- room 데이터는 24~48시간 후 자동 삭제
- 별도 회원가입/로그인 없음

---

## 4. 화면 구성

### 4.1 화면 목록

| 화면 | 설명 |
|------|------|
| 랜딩 | 서비스 소개 + 테스트 시작 버튼 |
| 질문 | 상황 설명 + 가능/불가능 선택 (N개 순차 진행) |
| 개인 결과 | 결과 타입 + 요약 문구 + 카카오톡 공유 + 커플 비교 버튼 |
| 대기 | A가 공유 후 B를 기다리는 화면 |
| 커플 비교 결과 | 두 사람 비교 요약 + 갈리는 포인트 + 각자 타입 |

### 4.2 진입 경로

```
[랜딩] → [질문] → [개인 결과]
                        ├── 카카오톡 공유 (개인 결과 링크)
                        └── 커플 비교
                              ├── room 생성 → 카카오톡 공유 (room 링크) → [대기]
                              └── room 링크로 진입 → [질문] → [커플 비교 결과]
```

## 5. API 명세 (개요)

| 엔드포인트 | 메서드 | 설명 |
|------------|--------|------|
| `/api/room` | POST | room 생성 + A 답변 저장, 세션 코드 반환 |
| `/api/room/:code` | GET | room 상태 조회 (B 답변 완료 여부) |
| `/api/room/:code/answer` | POST | B 답변 저장 |
| `/api/room/:code/result` | GET | 커플 비교 결과 조회 |

## 6. 결과 타입 시스템

### 6.1 타입 분류 기준
- 답변 패턴(가능 응답 비율 등)에 따라 결과 타입을 분류
- 타입별 고유 이름 + 설명 문구 + 이미지 제공

### 6.2 타입 예시 (추후 확정)
- **"철벽형"**: 대부분 불가능 → 이성 친구 사이 경계가 뚜렷한 타입
- **"오픈형"**: 대부분 가능 → 이성 친구 사이도 편하게 지내는 타입
- **"상황따라형"**: 혼합 → 상황에 따라 유연하게 판단하는 타입
- *(세부 타입은 기획 확정 시 추가)*

## 7. 카카오톡 공유

### 7.1 공유 유형

| 유형 | 내용 | 링크 |
|------|------|------|
| 개인 결과 공유 | "나의 여사친/남사친 경계 테스트 결과!" + 결과 타입 | 서비스 랜딩 페이지 |
| 커플 비교 초대 | "우리 경계 같을까? 테스트해보자!" | room 링크 (`?room={code}`) |

### 7.2 구현
- 카카오 JavaScript SDK 사용
- 커스텀 템플릿 메시지 (썸네일 + 타이틀 + 설명 + CTA 버튼)

## 8. 기술 고려사항

### 8.1 프론트엔드
- 모바일 퍼스트 반응형 웹
- 질문/결과 데이터는 클라이언트 번들에 포함 (정적)
- 상태 관리: 테스트 진행 상태는 클라이언트 로컬에서 관리

### 8.2 백엔드
- room CRUD만 담당하는 경량 API 서버
- room 데이터 TTL 설정 (24~48시간 자동 만료)
- 저장소: Redis 또는 TTL 지원 DB

### 8.3 인프라
- 정적 파일은 CDN 배포
- API 서버는 최소 스펙으로 운영 (room 관리만 담당)
- 커플 비교 트래픽 제한을 위한 rate limiting 설정

## 9. 성공 지표

| 지표 | 설명 |
|------|------|
| 테스트 완료율 | 테스트를 시작한 사용자 중 끝까지 완료한 비율 |
| 공유 전환율 | 결과를 본 사용자 중 카카오톡 공유를 누른 비율 |
| 커플 비교 전환율 | 결과를 본 사용자 중 커플 비교를 시작한 비율 |
| 커플 비교 완료율 | room 생성 후 B가 실제로 테스트를 완료한 비율 |
| 바이럴 계수 | 1명의 사용자가 유입시키는 평균 신규 사용자 수 |

## 10. 향후 확장 가능성
- 질문 세트 추가 (시즌별, 테마별)
- 친구 그룹 비교 (3인 이상)
- 결과 이미지 저장/다운로드
- 인스타그램 스토리 공유
- 익명 통계 대시보드 ("전체 사용자의 N%가 이 질문에서 갈렸어요")
